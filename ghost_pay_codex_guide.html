Build it with Codex

The day-by-day breakdown for two devs, formatted for use with OpenAI Codex. Includes how to set up the agent, how to prompt it at each stage, and how to stay in control of what it builds.

How to use Codex on this project

The iterative workflow

Codex is a cloud-based software engineering agent — it runs in a sandbox with your GitHub repo preloaded, writes code, runs commands, and proposes changes for you to review. The key mental model shift: treat it like a fast junior dev you review, not a magic button you trust blindly.

The core loop — repeat every task
  1. 01Give Codex one task at a time. Not the whole day, not two features. One scoped task. "Build the persistent notification with an action button that fires an Intent to open the overlay Activity."
  2. 02Read the diff before accepting. Codex shows you what changed. Read every file it touched. If you don't understand a line, ask Codex to explain it in the same session.
  3. 03Test it on the device before the next task. Never stack two Codex tasks without running the last one. Bugs compound and become impossible to trace.
  4. 04When it breaks, paste the full error into Codex. Don't paraphrase. Paste the exact terminal output or logcat. Codex fixes errors better with exact text than with descriptions.
  5. 05After it works, ask Codex to explain what it built. "Explain what this function does and why you structured it this way." This is how you learn the codebase as you go.
What Codex is bad at for this project
—Anything requiring a real connected device. It can't test mic capture, overlay rendering, or FCM delivery. You must test those manually.
—Broad vague prompts. "Build the backend" will produce something unfocused. "Write the /voice-ingest Firebase Cloud Function that accepts a multipart audio file, calls Whisper, and returns the JSON contract" produces something usable.
—Anything touching secrets or API keys. Never put real keys in prompts. Use environment variable names only. Codex writes the code; you fill in the values.
tip
Assign parallel tasks when the two dev repos are independent. Dev A and Dev B are building completely separate codebases until Day 2. Run Codex sessions simultaneously — one for the Nuxt/Capacitor repo, one for the Firebase repo. They don't block each other at all until the integration sync.
Setup — AGENTS.md

Write this before your first Codex session

AGENTS.md is a file you put in the root of each repo. Codex reads it before every task — it's your standing instructions. Without it, Codex guesses your conventions. With it, every output fits your project from day one. Create one for each repo.

Dev A repo — AGENTS.md
AGENTS.md
# Ghost Pay — Frontend / Native (Dev A) ## Stack - Nuxt 4 PWA mode with Capacitor for Android - Tailwind CSS for all styling - TypeScript throughout — no plain JS files - @capacitor/preferences for all local storage (never use localStorage) ## Design system - Background: #090919 - Accent / interactive: #00A693 (teal) - Typography: #FFFFFF - Font: Sora (already imported in nuxt.config) ## Android native rules - Entry point is a persistent foreground notification with an action button - The action button fires an Intent to launch OverlayActivity.kt - OverlayActivity is fullscreen and transparent — Capacitor WebView mounts inside it - FCM receiver is a BroadcastReceiver in Kotlin — do not implement in JS ## API contract - Voice payload POSTs to VOICE_INGEST_URL (env var) with Bearer JWT in Authorization header - Response shape is always: { status, payload: { kora_url, whatsapp_sent, parsed_data, audio_feedback_url } } - Store kora_url and audio_feedback_url via CapacitorPreferences after each success ## Testing - Do not write emulator tests for mic, overlay, or FCM — these must be tested on a physical device - Write unit tests for utility functions (JWT generation, response parsing) only ## Code style - Vue components use script setup syntax - No options API - Keep components under 150 lines — split if larger - All async calls wrapped in try/catch with explicit error state handling

# Ghost Pay — Backend / Orchestration (Dev B) ## Stack - Firebase Cloud Functions v2, TypeScript - Firestore for state (transactions, merchant profiles, FCM tokens) - Firebase Storage for YarnGPT audio files - No Express — use native Cloud Functions request/response ## Functions - /voice-ingest — main entry: audio → Whisper → Gemini → Kora → WhatsApp → FCM - /kora-webhook — receives Kora payment events, updates Firestore - /query-charge — fallback polling if webhook hasn't fired in 2 minutes ## API keys (always use env vars — never hardcode) - OPENAI_API_KEY (Whisper) - GEMINI_API_KEY - YARNGPT_API_KEY - KORA_SECRET_KEY - KORA_PUBLIC_KEY - WHATSAPP_TOKEN - WHATSAPP_PHONE_NUMBER_ID - JWT_SECRET ## JSON contract (frozen — do not change field names) Return shape from /voice-ingest: { status: "success", payload: { kora_url, whatsapp_sent, parsed_data: { amount, customer, item }, audio_feedback_url } } ## Gemini parsed intent shape (internal) { amount: number, description: string, customer_phone: string (E.164 format) } ## Security - All functions except /kora-webhook must verify JWT in Authorization header - /kora-webhook must verify Kora signature header before processing - Firestore rules: merchants read/write own data only ## Code style - Async/await throughout — no raw Promise chains - All external API calls wrapped in try/catch, log errors with structured logging - Functions should be under 80 lines — extract helpers if larger
Prompt patterns

How to write prompts that actually work

The single biggest reason Codex produces unusable code is a vague prompt. Here are the four patterns that cover every task in this sprint.

Pattern 1 — Build a new thing
template copy
Build [specific thing]. It should [behaviour 1], [behaviour 2]. Use [specific library/approach]. The function/component should accept [inputs] and return/emit [output]. Do not use [thing to avoid].
Pattern 2 — Fix a broken thing
template copy
This is broken. Here is the exact error: [paste full error / logcat / terminal output] Here is the relevant code: [paste the function or file] Fix it. Explain what was wrong before making the change.
Pattern 3 — Understand what Codex built
template copy
Explain what [function/file/component] does, step by step. Explain why you chose this approach over [alternative]. What would break if [specific part] was removed?
Pattern 4 — Integration check
template copy
Dev B's endpoint now returns this shape: [paste actual JSON response] Update [component/function] to consume this correctly. The field I need is [field name]. Handle the case where [field] is null or missing.
rule
Never start a prompt with "Can you..." Start with an imperative verb: Build, Write, Fix, Add, Update, Refactor, Explain. Codex responds to commands, not questions. Shorter prompts with clear verbs outperform long polite paragraphs.
